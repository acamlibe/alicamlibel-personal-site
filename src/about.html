As a Software Developer...

I believe that I am a very disciplined software developer. I continuously work towards achieving better code quality by challenging code I've written in the past and finding ways to improve upon it. Having known that I've written, to the best of my abilities, maintainable, performant, and clean code gives me personal satisfaction and confidence that the software will succeed.

My top software design philosophies (in order of importance):

  1. KISS (Keep it Simple, Stupid) - The question I frequently ask myself: can someone understand my code without extensive comments in my code or detailed documentation?
  2. Code Consistency - Code should be implemented in a consistent manner. Whether it's naming conventions or the architecture. A consistent codebase leads to better maintenance and reusability.
  3. Single Responsibility Principle - My father once told me, "If you can't describe what a class or function does in one sentence, it's doing too much."
  4. Performance - Imagine a Google search taking longer than 10 seconds, in this day and age. Systems should be as performant as can be, without sacrificing maintainability (unless it is absolutely necessary).

My top web design philosophies (in order of importance):

  1. Lightweight Page Load - Gone were the days of websites that served maybe a few hundred kilobytes worth of bandwidth to users. Nowadays, you go to Google.com and load ~3.4MB. Obviously with caching and the use of CDNs, it's not such a big deal, but I feel it has made developers too carefree.
  2. Semantic HTML - Semantic tags in HTML have made working with HTML so much more comprehensible. It also allows devices to better interpret webpages to translate to disabled users.
  3. Not everything has to be an SPA - JavaScript frameworks like Angular and React have taken the world by storm. Nowadays, knowing one of the two is almost a requirement for any web developer role. It's almost as if it's a standard now, but I believe that you need to look at your needs carefully. If you do not need a very dynamic page (such as the site you are on right now), then it is perfectly OK to serve HTML directly from a server.
  4. Design against your target audience - Using design systems such as Google's Material, Apple's Human Interface Guidelines, or Microsoft's Fluent is not a requirement for good design. Design should complement usability for your specific target audience. For example, in the energy sector many users are engineers or business experts, and often times they will favor higher density designs since they need to see a lot of data at once.
